# Characterising the ASCII layout algorithm

This folder bootstraps a **characterisation-test** suite for the layout
algorithm, in the sense of
[adewale/testing-best-practices](https://github.com/adewale/testing-best-practices):
pin what the code does today, with worked examples + property-based tests, so
the algorithm can be refactored safely and improved deliberately.

It contains three things:

| File | What it is |
|------|------------|
| [`contact-sheet.md`](./contact-sheet.md) | The **minimum set of worked examples** that, together, exercise every load-bearing decision in the layout. One phenomenon per example. (Generated by `scripts/characterization/contact-sheet.ts`.) |
| [`properties.md`](./properties.md) | The **smallest load-bearing set of properties** that characterise and determine the layout, with a minimality (mutation-kill) argument and academic grounding. |
| [`../../src/__tests__/characterization-layout.test.ts`](../../src/__tests__/characterization-layout.test.ts) | Those properties as executable `fast-check` property tests. |

No implementation code was changed to produce any of this.

---

## What "the layout algorithm" means here

This project has **two** distinct layout engines:

1. **ELK-based engine** (`src/layout-engine.ts`) — wraps Eclipse Layout Kernel's
   *layered* (Sugiyama) algorithm to produce pixel coordinates for **SVG/PNG**.
   The placement is a configured third-party black box; the bespoke part is the
   post-processing (layer snapping, edge bundling, margin routing, shape
   clipping).
2. **ASCII grid engine** (`src/ascii/grid.ts` + `pathfinder.ts` +
   `edge-routing.ts` + `edge-bundling.ts`) — a **hand-written** grid layout with
   **A\* orthogonal edge routing** that renders flowcharts and state diagrams as
   character art. It is a TypeScript port and extension of
   [`AlexanderGrooff/mermaid-ascii`](https://github.com/AlexanderGrooff/mermaid-ascii).

**This characterisation targets the ASCII grid engine.** It is the genuinely
"ours" algorithm (the ELK one delegates placement), it is pure / synchronous /
deterministic / dependency-free, and it is the surface the project already
mutation-tests (`stryker.ascii.config.json`). It is therefore the ideal
characterisation subject. The flowchart and `stateDiagram-v2` families share
this exact pipeline (`src/ascii/index.ts`); the other families (sequence, class,
ER, pie, quadrant, xychart, timeline, journey, architecture) have their own
renderers and are out of scope here.

### The pipeline in one line

```
parseMermaid → convertToAsciiGraph → createMapping (grid + A*) → drawGraph → canvasToString
```

`createMapping` (`grid.ts`) is the heart: it places each node as a **3×3 grid
block**, assigns layers along the flow axis with a **stride of 4**, routes every
edge with a **bounded, deterministic A\***, and bundles fan-in/fan-out edges onto
shared trunks.

## How the layout works (the load-bearing mechanics)

- **Grid model.** Every node owns a 3×3 block of an integer grid; `gridCoord` is
  its top-left cell, and the eight surrounding cells are edge attachment points.
  Blocks are placed 4 grid units apart (3 cells + a 1-cell routing gutter).
- **Layer assignment.** Roots (nodes with no incoming edge) go at level 0;
  each child is placed one level (stride 4) along the **flow axis** (y for
  TD/BT, x for LR/RL). TD and LR are the same code with x/y transposed; **BT =
  TD then vertical flip**; **RL is currently treated as LR**.
- **A\* routing.** Edges are routed cell-by-cell with A\* over the same grid that
  tracks node occupancy. Moves are 4-directional only (⇒ **orthogonal**, no
  diagonals); the heuristic is Manhattan distance + a corner penalty (prefers
  straight runs); ties break FIFO (⇒ **deterministic**); the search is bounded
  (⇒ **terminates**, returns `null` + a straight fallback if walled off).
- **Edge bundling.** Sibling fan-out / fan-in edges of the same style merge onto
  a single shared trunk that splits at one junction; labelled siblings (which
  can't bundle) share a trunk prefix instead.
- **Best-effort, not a solver.** For cyclic / dense graphs the multi-pass
  placement can drop or overlap nodes. That limitation is real, current, and
  pinned by property **P10** rather than hidden.

A fuller mechanical anatomy (with `file:line` citations for all 30+ invariants)
informed `properties.md`; the citations are inlined there.

## How this connects to the testing literature

- **Characterisation tests** (Feathers, *WELC* ch. 13): tests that document the
  system's *actual* behaviour, not its intended behaviour — the safety net for
  refactoring. The contact sheet is the worked-example form; the property suite
  is the generalised form.
- **Approval / golden tests** (Falco): the contact sheet is generated and
  reviewed as a diff — a surprising change is a regression, an expected change
  is the new approved baseline. The repo's `src/__tests__/testdata/` corpus is
  the byte-level golden layer beneath it.
- **Property-based testing** (fast-check): the kernel asserts **invariants** and
  **metamorphic relations** rather than exact outputs, so it generalises across
  thousands of generated graphs instead of a handful of fixtures.
- **Mutation testing** (Stryker, already configured): the oracle for *which*
  properties are load-bearing — a property is kept only if it kills a mutant no
  other test kills. See the minimality argument in `properties.md`.

Together: **goldens characterise, properties generalise, mutation proves which
assertions do work.**

## Reproduce

```bash
bun install

# Executable characterisation kernel (13 properties, ~2000 assertions)
bun test src/__tests__/characterization-layout.test.ts

# Regenerate the contact sheet, then review the diff and commit
bun run scripts/characterization/contact-sheet.ts
git diff docs/layout-characterization/contact-sheet.md

# (optional) confirm the properties are load-bearing
npx stryker run stryker.ascii.config.json
```
