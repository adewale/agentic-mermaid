# Product

## Register

brand

## Users

Agentic Mermaid serves engineers, documentation authors, and coding agents that need deterministic Mermaid rendering, local verification, and safe structured diagram edits. Website visitors usually arrive with one of four jobs: try the browser editor, install/use the toolkit locally, wire it into an agent harness or MCP client, or inspect supported diagram families and docs before adopting it.

## Product Purpose

The website explains and demonstrates Agentic Mermaid as a local-first, agent-native Mermaid runtime. Success means a visitor quickly understands the product boundary, chooses the right path for their intent, and trusts the claims because the UI exposes concrete routes, examples, machine-readable surfaces, and honest constraints.

## Brand Personality

Precise, calm, and technically rigorous. The interface should feel like a well-typeset manual attached to a capable workbench: sparse, deterministic, practical, and confident without hype.

## Anti-references

Do not make this look like a generic SaaS landing page with card grids, glass panels, glowing AI gradients, loud hero metrics, or overbearing CTAs. Do not imply hosted Code Mode, arbitrary execution, or a production render API. Avoid decorative motion, vague “AI magic” language, consumer-soft pills, and inconsistent app-vs-doc navigation.

## Design Principles

1. Lead every visitor to the next honest action: try the editor, install locally, wire an agent, or read capabilities.
2. Make local-first boundaries visible before trust is asked for; hosted execution stays explicitly out of scope.
3. Practice the product’s determinism: consistent routes, labels, CTA language, and machine-readable mirrors.
4. Keep the document-first voice, but use stronger affordances where a user must choose or act.
5. Show real renderer output and concrete commands instead of decorative claims.

## Accessibility & Inclusion

Target WCAG AA for text and controls. Preserve keyboard-visible focus, readable contrast in every theme, reduced-motion alternatives, semantic landmarks, and touch targets around 40px where practical. The editor must support keyboard users, screen-reader-friendly state changes, copy/export affordances, and understandable error states.
