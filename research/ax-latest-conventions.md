# Research: 2025–2026 AX conventions for static docs + browser editor

## Summary
Current accessibility practice for a static documentation site with a browser editor is still anchored in WCAG 2.2 Level AA, native HTML semantics first, APG patterns only for custom widgets, and explicit respect for user preferences such as reduced motion and forced colors. For this repo, the most important conventions are: preserve semantic documents/tables/code, make custom menus/dialogs truly keyboard-operable, provide visible focus and 24×24+ targets, avoid color-only status, honor reduced motion/high contrast, and give generated diagrams/text outputs accessible names/descriptions.

## Research angles
- **Standards baseline:** WCAG 2.2, ARIA/APG, HTML living standard for `inert`, `popover`, focus, dialogs, and semantic elements.
- **Editor/product behaviors:** keyboard shortcuts, roving focus, menus, copy feedback, modal/popover dismissal, local status messaging.
- **Visual preferences:** target size, focus appearance, reduced motion, forced colors/high contrast, color contrast, non-text contrast.
- **Repo applicability:** static mockups/build pipeline in `mockups/*.html`, `mockups/styles.css`, `mockups/theme.js`, and generated site transforms in `website/build.ts`.

## Findings
1. **WCAG 2.2 AA is the right baseline; 2.2 adds concrete UI obligations beyond contrast.** WCAG 2.2 includes new success criteria such as Focus Not Obscured (2.4.11/2.4.12), Focus Appearance (2.4.13), Dragging Movements (2.5.7), Target Size Minimum (2.5.8), Consistent Help (3.2.6), and Accessible Authentication (3.3.8/3.3.9). For this repo, Focus Not Obscured, Focus Appearance, Target Size, and Dragging Movements are directly relevant to nav, theme menu, editor panes, splitters, canvas interactions, and diagram manipulation. [WCAG 2.2](https://www.w3.org/TR/WCAG22/)
2. **Focus indicators must be intentionally designed, not browser-default afterthoughts.** WCAG 2.2 Focus Appearance AA expects a clearly visible indicator with sufficient area and contrast; product convention is a consistent 2px outline/ring with offset, not just color change. `mockups/styles.css` already defines `a:focus-visible, .btn:focus-visible, .iconbtn:focus-visible, button:focus-visible { outline: 2px solid var(--link); outline-offset: 2px; }`, but it should be verified across every theme and against forced-colors. [WCAG Understanding Focus Appearance](https://www.w3.org/WAI/WCAG22/Understanding/focus-appearance.html)
3. **Target size has moved from “nice to have” to a WCAG 2.2 AA check.** Pointer targets should be at least 24×24 CSS px unless an exception applies; for touch/editor UI, practical product expectation remains 40–44px minimum because dense toolbars and panes are error-prone. The repo’s `--control-h: 40px` is aligned; mobile masthead links at `min-height: 36px` are likely acceptable but should not be used for icon-only editor controls. [WCAG Understanding Target Size Minimum](https://www.w3.org/WAI/WCAG22/Understanding/target-size-minimum.html)
4. **Custom menus should follow APG behavior or use simpler native alternatives.** The theme picker uses `button[aria-haspopup][aria-expanded]`, a `role="menu"`, `menuitemradio`, roving `tabIndex`, Escape, arrows, Home/End, and `inert` while closed in `mockups/theme.js`; this is close to APG menu-button conventions. Residual issue: APG menus are for application-style command menus; if this is ordinary settings selection, a native `<select>` or listbox/radio-group pattern may be simpler and more predictable for assistive tech. [WAI-ARIA APG Menu Button Pattern](https://www.w3.org/WAI/ARIA/apg/patterns/menu-button/), [WAI-ARIA APG Radio Group Pattern](https://www.w3.org/WAI/ARIA/apg/patterns/radio/)
5. **Use `inert`/`popover`/`dialog` semantics to avoid hidden focus traps.** HTML `inert` removes a subtree from sequential focus navigation and accessibility hit-testing; `popover` provides top-layer display/light-dismiss behavior without pretending to be modal. The existing theme menu sets `menu.inert = true` while closed, which is good. Future editor overlays should prefer `popover` for non-modal menus/tooltips and `<dialog>` for true modal flows, with focus restoration. [MDN inert](https://developer.mozilla.org/en-US/docs/Web/HTML/Global_attributes/inert), [MDN Popover API](https://developer.mozilla.org/en-US/docs/Web/API/Popover_API), [HTML popover attribute](https://html.spec.whatwg.org/multipage/popover.html)
6. **Reduced motion applies to transitions, view transitions, and animated diagram/editor effects.** The repo already disables view-transition animation and global animation/transition duration under `prefers-reduced-motion: reduce` in `mockups/styles.css`, and `mockups/theme.js` bypasses View Transitions when reduced motion matches. Keep that pattern for editor previews, auto-layout animations, scroll-to-error behavior, and diagram transitions. [WCAG Understanding Animation from Interactions](https://www.w3.org/WAI/WCAG22/Understanding/animation-from-interactions.html), [MDN prefers-reduced-motion](https://developer.mozilla.org/en-US/docs/Web/CSS/@media/prefers-reduced-motion)
7. **Forced colors/high contrast needs explicit CSS treatment.** In forced-colors mode, subtle `color-mix()`, shadows, background images/noise, and theme-dependent contrast may be overridden or disappear. Add `@media (forced-colors: active)` rules for controls, links, skip link, focus outlines, diagrams, tables, and popovers using system colors (`Canvas`, `CanvasText`, `LinkText`, `ButtonText`, `Highlight`) and avoid `forced-color-adjust: none` except for carefully tested brand/diagram cases. [CSS Color Adjustment Module: forced colors](https://drafts.csswg.org/css-color-adjust-1/#forced), [MDN forced-colors](https://developer.mozilla.org/en-US/docs/Web/CSS/@media/forced-colors)
8. **Accessible names must come from stable text or ARIA, especially for icon/canvas/SVG UI.** The GitHub link in `mockups/editor.html` has `aria-label="GitHub repository (external)"`, which is appropriate. The brand mark is currently visual; if it is a link/control, it needs a text name, and decorative inner canvas/SVG should be `aria-hidden="true"`. Diagram SVGs should expose a title/description or be referenced by adjacent caption text depending on whether the SVG conveys unique content. [AccName 1.2](https://www.w3.org/TR/accname-1.2/), [W3C Images Tutorial](https://www.w3.org/WAI/tutorials/images/)
9. **Semantic tables should preserve header relationships through responsive layouts.** The verify table in `mockups/editor.html` uses `<th>` but lacks `<thead>/<tbody>` and `<caption>`; adding these improves navigation and context. The `.warning-table` mobile pattern hides `thead` visually and injects `td::before` from `data-label`; ensure the real headers remain in the accessibility tree rather than `display:none` if used for meaningful data. [W3C Tables Tutorial](https://www.w3.org/WAI/tutorials/tables/), [HTML table model](https://html.spec.whatwg.org/multipage/tables.html)
10. **Async product feedback should be visible and announced.** Copy buttons in `mockups/theme.js` update text to “Copied” or “Copy failed”, which is visible and good; for future parse/render/verify actions, use a persistent status region such as `role="status"`/`aria-live="polite"` for success/progress and `role="alert"` sparingly for urgent failures. Do not encode warnings only in color; pair counts/colors with text and icons. [WAI-ARIA live regions](https://developer.mozilla.org/en-US/docs/Web/Accessibility/ARIA/ARIA_Live_Regions), [WCAG Use of Color](https://www.w3.org/WAI/WCAG22/Understanding/use-of-color.html)
11. **Keyboard operation for editor workflows must include non-pointer alternatives for dragging/resizing.** WCAG 2.2 Dragging Movements requires single-pointer alternatives for drag interactions; practical editor convention also requires keyboard alternatives for pane resizing, node selection, zoom, pan, and command invocation. This is especially relevant if the browser editor later supports direct diagram manipulation. [WCAG Understanding Dragging Movements](https://www.w3.org/WAI/WCAG22/Understanding/dragging-movements.html)
12. **Practical UI quality expectations from design-skill references align with accessibility: crisp hierarchy, restrained motion, direct manipulation with fallback, and tactile feedback.** The requested skill URLs could not be live-fetched in this runtime, but their likely themes—motion vocabulary, “feel better” interaction polish, and impeccable UI craft—should be constrained by AX rules: use motion as state communication only when reduced-motion-safe, keep targets generous, make feedback textual and announced, and avoid decorative polish that reduces contrast or keyboard clarity. Requested refs: [emilkowal.ski/skill](https://emilkowal.ski/skill), [Emil design eng skill](https://www.skills.sh/emilkowalski/skill/emil-design-eng), [Make interfaces feel better](https://www.skills.sh/jakubkrehel/make-interfaces-feel-better/make-interfaces-feel-better), [Impeccable](https://www.skills.sh/pbakaus/impeccable), [animations.dev vocabulary](https://animations.dev/vocabulary)

## Repo-specific review findings
1. **High — Add forced-colors coverage.** `mockups/styles.css` has robust theme tokens and reduced-motion rules but no observed `@media (forced-colors: active)` block. Add system-color rules for focus outlines, links, buttons, theme menu, table borders, code panels, skip link, and diagram plates.
2. **High — Ensure generated diagrams have accessible text alternatives.** `mockups/editor.html` embeds an SVG diagram with visible text but no `<title>`, `<desc>`, `role="img"`, or `aria-labelledby`. For static docs, either treat as image with title/desc/caption association, or mark decorative only if the code/text output fully duplicates it.
3. **High — Keep custom theme menu APG-consistent or simplify pattern.** `mockups/theme.js` is mostly keyboard-complete, but `role="menu"`/`menuitemradio` is a high-ceremony app-menu pattern. Consider `popover` + radio group/listbox if the control is settings rather than commands.
4. **Medium — Strengthen semantic tables.** `mockups/editor.html` verify table should add `<caption>`, `<thead>`, `<tbody>`, and scoped headers (`<th scope="col">`/`scope="row"`) if used beyond decorative sample content.
5. **Medium — Audit focus indicator contrast across all themes.** `mockups/styles.css` uses `outline: 2px solid var(--link)`; some theme link colors are adjusted, but Focus Appearance should be measured in paper/dusk/solarized/nord/catppuccin and forced-colors.
6. **Medium — Mobile nav targets should stay product-consistent.** `mockups/styles.css` mobile masthead links drop to 36px; acceptable for text links, but future icon buttons/toolbars should keep 40–44px.
7. **Low/Medium — Copy feedback should be announced when async.** `mockups/theme.js` changes button text after copy; for pages where copy success matters, add `aria-live="polite"` or `role="status"` associated with copy controls.
